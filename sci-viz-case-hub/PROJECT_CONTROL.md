# Sci-Viz Case Hub 项目总控文档

## 1. 项目定位

Sci-Viz Case Hub 是一个**科研视觉案例自动收集、分类、复核与沉淀工具**。

它不是普通图片收藏夹，也不是单一摄影图库，而是用于科研视觉调研和工作室知识资产建设的内部工具。系统需要帮助用户从网页、截图、本地上传和后续自动采集来源中收集科研相关视觉材料，并通过 OCR、AI 分类和人工复核，将零散素材整理成可追溯、可分类、可评分、可导出的科研视觉案例库。

项目服务于科研影像工作室的前期调研、视觉基准分析、风格图谱构建、案例拆解、视觉策略制定和方法论沉淀。

---

## 2. 项目核心目标

系统的核心目标不是"保存更多图片"，而是把科研相关视觉材料转化为**结构化案例**。

核心流程为：

```text
采集 → 保存 → OCR识别 → AI分类 → 入库 → 人工复核 → 评分/备注 → 导出
```

所有功能设计和技术实现都应围绕这条主流程展开。

系统必须覆盖广义的科研视觉表达，不应只围绕摄影设计。

```text

实验过程图
论文机制图
实验流程图

期刊封面
科普图解
混合媒介图像

```

后续扩展时，应继续保持"科研视觉表达"的广义范围，不能退化为普通图片图库。

## 4. 产品形态

系统应被设计为**科研视觉调研工作台**，而不是图片墙。

界面和功能应帮助用户完成：

```text
发现案例
确认来源
理解图像内容
判断视觉类型
识别可借鉴点
人工复核
评分和备注
沉淀为高价值案例
导出用于报告、风格图谱或案例拆解
```

产品体验应强调：

```text
来源清楚
分类明确
复核高效
状态可见
评分直观
导出可用
```

## 5. 数据存储原则

系统应区分**文件本体**和**结构化元数据**。

文件本体包括：

```text
图片
截图
PDF
视频
缩略图
导出文件
```

结构化元数据包括：

```text
来源链接
来源域名
网页标题
图片链接
OCR文本
AI分类结果
AI摘要
可借鉴点
风险提示
人工评分
人工备注
复核状态
创建时间
更新时间
```

原则：

```text
数据库存元数据
文件存本地 uploads 或对象存储
不要把大文件直接塞进数据库
远程图片必须保存本地或云端副本，不能只保存原始网页 URL
```

## 6. 当前推荐部署与协作方案

系统作为工作室内部工具使用，不需要公开发布，不需要域名和备案。

推荐方案：

```text
国内轻量服务器：运行前端、后端和 SQLite 数据库
阿里云 OSS：存图片、截图、PDF、视频、缩略图等文件
SQLite：存案例元数据、分类、OCR、AI分析、评分、备注和复核状态
简单登录：保护后台访问
```

协作者通过浏览器访问同一个内部后台，可以查看、修改、复核、评分和导出案例。

当前不需要 PostgreSQL。
只有在多人高频同时编辑、数据量显著增大、权限复杂化或 SQLite 明显不够用时，再考虑迁移 PostgreSQL。

## 8. 自动采集原则

系统支持从用户指定的网页 URL 中自动提取候选图片。

自动采集的合理模式是：

```text
用户提供明确 URL
→ 系统访问公开网页
→ 提取页面图片和上下文
→ 过滤 logo、icon、广告、小图等无效图片
→ 保存候选图片
→ 进入 OCR 和 AI 分类流程
→ 等待人工复核
```

自动采集不是全网爬虫，也不是无边界抓取。
系统应优先支持**指定 URL 批量采集**，后续可扩展为来源管理、定时采集或关键词采集。

自动采集必须保留：

```text
source_url
source_domain
page_title
image_url
alt_text
context_text
capture_type = crawler
```

## 9. AI 与人工判断关系

AI 的角色是辅助整理，而不是最终决策。

AI 可以负责：

```text
初步分类
OCR辅助理解
生成摘要
提炼可借鉴点
提示风险
给出置信度
```

但用户必须可以人工修改：

```text
分类
标签
状态
评分
备注
是否批准
是否拒绝
是否重新分析
```

AI 输出不能直接作为最终结论。

## 11. 复核状态

案例状态建议包括：

```text
pending_ai_analysis
needs_review
low_confidence_review
approved
rejected
analysis_failed
```

含义：

```text
pending_ai_analysis：已保存，等待 AI 分析
needs_review：AI 已完成分析，等待人工复核
low_confidence_review：AI 置信度低，需要重点复核
approved：人工确认保留
rejected：人工确认不保留
analysis_failed：OCR 或 AI 分析失败
```

原则：

```text
OCR 或 AI 失败时，案例仍应保存
失败案例应允许重试或人工处理
AI 分析完成后仍需人工确认
```

## 12. 评分规则

评分用于判断案例价值。

建议规则：

```text
1 分：无参考价值
2 分：普通参考
3 分：可保留
4 分：值得拆解
5 分：标杆案例
```

4–5 分案例可作为后续"高价值案例""案例拆解""风格图谱素材""报告输出"的重点来源。

## 13. 采集目标与分类 KPI

系统的自动采集不应以"尽可能多抓图片"为目标，而应以建设一个覆盖均衡、来源可靠、可用于科研视觉策略分析的结构化案例库为目标。

采集应围绕以下原则：

```text
按来源分级
按年份范围筛选
按分类维度设定 KPI
按候选价值评分排序
达到 KPI 后停止或降级采集
低价值和重复图片不应持续入库
```

### 13.1 采集范围

案例库应同时覆盖：

```text
近年案例：优先覆盖 2020 年至今，用于判断当前科研传播和视觉表达趋势
经典案例：不限年份，但必须来自高价值来源、封面档案、精选图库、年度图片、获奖图片、研究亮点或媒体图库
```

默认比例建议：

```text
近年案例：约 70%
经典案例：约 30%
```

### 13.2 分类 KPI 原则

每个核心分类维度都应有目标数量。系统发现候选图片后，应先判断它可能补足哪个分类缺口，再决定是否采集和入库。

重点 KPI 维度包括：

```text
media_type
content_type
discipline
visual_style
rating
source_domain / source_type
```

当某一类别达到目标数量后：

```text
普通候选图停止采集
高价值候选图仍可少量进入复核
重复图、低分候选图、无上下文图片直接跳过
```

### 13.3 第一阶段建议目标

第一阶段不追求大规模图片库，而是建立可人工消化、可用于案例拆解的高质量候选集。

建议目标：

```text
候选入库总量：1000–3000 张
人工保留案例：300–800 张
高价值案例（rating 4–5）：50–150 张
```

**当前进度（2026-05-29）：**

```text
总入库案例：3046 张（已达第一阶段上限3000）
├── 中国科学院 (CAS)        73 张 · A级
├── 清华大学               123 张 · S级
├── 北京大学               143 张 · S级
├── 复旦大学               106 张 · A级
├── 上海交通大学            97 张 · A级
├── 中国科学技术大学        51 张 · A级
├── Cell Research          30 张 · S级（Playwright）
├── Light: SA              62 张 · S级（Playwright）
└── 其他来源              2361 张
```

各来源可按价值分级设置配额：

```text
S 级来源：200–500 张候选
A 级来源：80–200 张候选
B 级来源：30–80 张候选
```

每个候选图片应计算 collection_score，用于决定是否采集：

```text
来源权重
发布时间
图片尺寸
是否文章主图
是否 figure / figcaption 图片
是否有 alt / caption / 上下文
是否来自 cover / gallery / feature / research / story 页面
是否补足当前分类 KPI 缺口
是否为完全重复图片
```

自动入库去重只应跳过文件哈希完全一致的图片。相似图片、同一主题不同角度、同一实验不同裁切、同一设备不同场景，都应保留进入人工复核，不应被系统自动删除。

### 13.4 分类归一化要求

AI 输出和爬虫来源可能产生非标准分类，例如组合分类、细分学科或临时描述。入库前应尽量映射回核心分类体系。

原则：

```text
数据库统计和 KPI 判断必须使用标准分类
组合分类应拆分或映射到主分类
无法确定时归入"不确定"，等待人工复核
不要让临时分类值污染 KPI 统计
```

## 14. UI/UX 方向

系统界面应是高级、简约、专业的科研视觉工作台，而不是普通图库。

风格参考方向：

```text
Linear：高级工具感和清晰状态管理
Raycast：高效、轻量、低干扰操作
Vercel / shadcn admin：简约后台组件质感
Airtable / Notion：结构化资料库管理
Label Studio：复核与标注流程
```

视觉原则：

```text
低饱和
细边框
轻阴影
清晰层级
信息密度适中
状态 badge 清楚
来源域名明显
复核操作高效
```

核心页面应围绕：

```text
工作台
案例库
待复核
高价值案例
自动采集
导出
设置
```

## 15. 内部协作与安全原则

系统作为内部工具使用，也需要基本安全保护。

至少应支持：

```text
简单登录
管理员/协作者角色
API key 不暴露到前端
服务器环境变量管理
SQLite 数据库备份
OSS 文件访问控制
```

建议角色：

```text
admin：可配置、删除、导出、复核
editor：可查看、编辑分类、评分、备注、复核
```

不需要一开始做复杂组织、多租户、邀请系统，但不能让后台裸奔公开访问。

## 16. 开发助手工作规则

任何 AI / Codex / Cursor / Claude Code 在处理本项目时，应遵守：

```text
先读取 PROJECT_CONTROL.md
再读取 docs/README.md
再读取 docs/ARCHITECTURE_AND_DELEGATION.md
先读取项目文件，再判断当前实现
修改前先检查现有代码和接口
优先保护主流程，不要破坏采集、保存、分类、复核、导出
优先小步修改，避免不必要的大规模重构
新增字段或功能时尽量保持向后兼容
涉及 OCR、AI 分类、文件存储、数据库结构、部署方式时，必须说明影响范围
每次修改后说明：改了哪些文件、解决了什么问题、如何验证、是否有遗留问题
如果需求存在歧义，围绕"科研视觉案例结构化管理"这一核心目标判断
```

高级工程负责采集架构、数据模型、去重、安全下载、KPI/评分算法、存储抽象、登录部署、反爬降级策略等地基工作。

低成本 AI 可以承接 selector 补充、前端列表展示、文档维护、dry-run 记录、只读统计脚本等低风险任务，但不得绕过去重、安全下载、分类归一化和采集边界。

## 17. 最重要的判断标准

所有产品和技术决策都应围绕这一句话：

```text
Sci-Viz Case Hub 的目标不是保存更多图片，而是把科研相关视觉材料转化为可追溯、可分类、可复核、可评分、可导出的结构化案例库。
```

如果一个功能有助于这个目标，可以纳入迭代；如果偏离这个目标，应谨慎处理。
